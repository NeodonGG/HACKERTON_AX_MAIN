## [1단계] 날짜 형식 통일
- 결정 내용: `2026-05-02`(ISO) / `2026/05/06`(슬래시) / `5월 4일`(한글) 3종 혼용 형식을 모두 `YYYY-MM-DD` ISO 형식으로 통일. 한글 표기(`5월 4일` 등)는 연도 정보가 없어, 데이터 전체가 2026년 5~6월 구간에 몰려 있는 문맥을 근거로 2026년으로 보완.
- 선택한 이유: 정렬·집계·차트 처리에 일관된 포맷이 필요하고, ISO 형식이 원본에서 가장 많이 쓰인 표준 형식이므로 이를 기준으로 통일.
- 대안으로 고려했던 것: 원본 형식을 그대로 두고 파싱 시점에만 변환하는 방법도 있었으나, 재현성(파이프라인 재사용) 관점에서 저장 단계에서 통일하는 쪽이 더 안전하다고 판단.

  - **칭찬**: 서비스·담당자 대응·산출물에 대한 긍정적 평가·감사 표현
  - **일반 문의**: 불만·요청·칭찬 요소 없이 순수하게 정보·절차·기준을 확인하는 질문
- 선택한 이유: "불만 vs 기능요청" 경계가 가장 모호한데, "문제 삼는 대상이 이미 존재하는 기능인가, 아직 없는 기능인가"를 기준으로 삼으면 판단이 일관되고 재현 가능함.

## [3단계] 파이프라인 구축 및 원본 데이터 검증
- 결정 내용: `pipeline/generate_report.py`를 작성해 CSV 입력 -> 날짜 정제(3형식 통일) -> 규칙 기반(키워드 매칭) 분류 -> 집계 -> `voc-report.md`/`voc-report.html` 자동 생성을 구현. 원본 `data/voc.csv`로 실행해 2단계 수동 분류(정답)와 대조한 결과 **44건 중 43건 일치 (97.7%)**.
- 유일한 불일치: id 038("카본링크 모바일 앱이 없어서 불편합니다")을 파이프라인은 "불편합니다" 키워드로 **불만** 분류. 수동 분류는 "존재하지 않는 기능(모바일 앱)의 신설 필요성"으로 보아 **기능 요청**으로 분류. "기능이 아예 없는지 vs 있는데 불편한지"는 순수 키워드 매칭으로 구분 불가능한 근본적 한계로 판단, 임의 보정하지 않고 그대로 기록.
- 발견·수정한 버그 2건(수정 전 44건 중 40건/90.9% 일치):
  1) "감사" 키워드가 "감사합니다"(고마움)와 "내부 감사/감사법인"(audit, 무관한 의미)에 모두 매칭되어 id 015, 030을 칭찬으로 오분류 -> 키워드를 "감사합니다"/"감사드립니다"로 구체화하여 수정.
  2) "필요합니다" 키워드가 "기능이 필요합니다"뿐 아니라 "긴급 대응이 필요합니다"(id 045)에도 매칭되어 기능요청으로 오분류 -> "기능이 필요"/"서비스가 필요"로 구체화하여 수정.
- 선택한 이유: Standard 요건("새 CSV 넣어도 정제→분류→집계→리포트 동일 품질로 자동 생성")을 만족하려면 결정론적·재현 가능한 규칙 기반 로직이 필요. 완벽한 자연어 이해 없이 키워드만으로 100% 일치는 기대하기 어려우므로, 실제 검증 수치(97.7%)와 한계(id 038 사례)를 투명하게 리포트에 명시하는 것이 임의로 100%라고 주장하는 것보다 신뢰도 있는 접근.
- 대안으로 고려했던 것: id 038 사례에 맞춰 "불편합니다"를 복잡한 예외 규칙으로 처리하는 방법도 검토했으나, 특정 사례에 맞춘 과적합(overfitting)이라 판단해 기각. 대신 한계를 리포트에 명시하는 쪽을 선택.

## [4단계] 더미 데이터 생성

- **결정 내용**: `scripts/generate_dummy_voc.js`(Node.js, 시드 고정 20260701)로 `data/voc_dummy.csv`(200건)를 생성. 컬럼 구조(`id,date,channel,customer_type,content,keyword_hint`)는 원본 `data/voc.csv`와 동일하게 유지.
- **규모 산정 근거**: 원본 44건의 약 4.5배인 200건으로 설정 (PLAN.md 권장 범위 150~300건의 중간값). 5단계 파이프라인 재현성 검증에 쓸 만큼 크면서도 육안 검증이 가능한 규모로 판단.
- **유형 분포 산정 근거**: 2단계에서 확정한 원본 44건 최종 분류 집계(불만 9 / 기능 요청 12 / 칭찬 8 / 일반 문의 15 → 20.5% / 27.3% / 18.2% / 34.1%)를 그대로 200건에 적용해 목표 분포로 설정.
- **파이프라인 키워드 스템과의 정합성 확보**: `pipeline/generate_report.py`의 `classify()`가 사용하는 정확한 키워드 스템(PRAISE_STEMS: 감사합니다/감사드립니다/덕분/좋았/만족/인상적/올라갔/편하게/줄었습니다, COMPLAINT_STEMS: 오류/반려/끊깁니다/차이가 납니다/제한적/누락됩니다/불편합니다/해결해주세요 등, REQUEST_STEMS: 좋겠/기능이 필요/서비스가 필요/제공해/만들어/지원해줄/추가해/템플릿)에 실제로 매칭되도록 유형별 문장 템플릿을 설계. 3단계에서 스템을 구체화한 버그 수정(예: "감사"→"감사합니다", "필요합니다"→"기능이 필요") 이후 버전에 맞춰 템플릿을 정렬함.
- **검증 결과**: `pipeline/generate_report.py`를 `data/voc_dummy.csv`에 실제로 실행 → 200건 중 196건 처리(중복 4건 자동 제외), 분류 결과 불만 22.4% / 기능 요청 27.0% / 칭찬 17.3% / 일반 문의 33.2%로 목표 분포(20.5/27.3/18.2/34.1%)와 오차 2%p 이내로 근접. `voc-report.md`/`voc-report.html` 모두 오류 없이 생성됨을 확인.
- **원본 데이터 이슈 재현**:
  - 날짜 형식 3종 혼용(ISO 70% / 슬래시 17% / 한글 "N월 N일" 13%), 날짜 범위는 2026-05 ~ 2026-08 약 3개월로 수렴.
  - 채널 결측 9건(약 4.4%, 200건 기준) — 원본 45건 중 2건 결측 비율을 동일하게 적용.
  - 완전 중복행 2쌍을 의도적으로 삽입(id만 다르고 날짜·채널·내용 전부 동일, 원본 id001=id031 사례 재현) + 템플릿·용어 조합의 우연한 중복 2건이 추가로 발생 → 총 4건(2.0%)이 파이프라인에서 자동 제외됨. 원본 중복률(약 2.2%)과 유사한 수준.
  - 경계 케이스(불만+기능요청 혼재) 3건 — 원본 id043(보고서 커스터마이징 제한 불만 + 개선요청 혼재, 최종 "불만"으로 분류)처럼 애매한 문장을 삽입. `COMPLAINT_STEMS`("제한적", "불편합니다")를 포함시켜 파이프라인에서도 3건 모두 "불만"으로 수렴하도록 설계 — 2단계 id043 처리 근거와 일치.
- **선택한 이유**: 스크립트 기반 생성으로 재현성(고정 시드) 확보, 이후 파이프라인 재검증 시 동일 결과 재생성 가능. 초기에는 템플릿×용어 조합만 사용했으나 200건 규모에서 우연한 완전 중복이 30건(15%)까지 발생해 원본보다 비현실적으로 높아짐 → 문두 도입구(OPENERS) 8종을 추가해 조합 공간을 넓혀 우연한 중복을 4건(2%) 수준으로 낮춤. Python 대신 Node.js 사용(로컬 환경의 python 실행 별칭 문제 회피, 실행 확인됨).
- **대안으로 고려했던 것**: (1) 원본 44건을 단순 반복 복제 — 분포·문체 다양성이 없어 파이프라인 검증 가치가 낮다고 판단해 배제. (2) 완전 무작위 문장 생성 — CBAM/CSRD/Scope 등 원본 용어 맥락 및 파이프라인 키워드 스템과 무관해질 위험이 있어 배제. 대신 원본 문체를 반영하고 실제 분류기 스템에 맞춘 템플릿+용어 슬롯 방식을 채택.
- **투명성 명시**: 이 더미 데이터는 원본을 대체하지 않으며, Basic 단계의 분류표·집계·인사이트는 원본 44건 기준으로만 작성함 (`PLAN.md` 범위 확정 사항 1항 참조). 더미 데이터는 Standard의 파이프라인 재현성 검증과 Challenge 기획안의 대규모 빈도 보조 근거로만 사용.

## [7단계] Challenge 우선순위 프레임워크 및 제품 개선 기획안
- 결정 내용: 우선순위 판단 기준을 "비즈니스 임팩트 45% + 규제 긴급성 25% + VoC 빈도 30%"의 가중합(100점, 85점 이상 P0/70~84 P1)으로 직접 설계. 이 기준으로 기획안 2건을 산출: ①CBAM 신고 기한 알림+비용 추정/제출 현황 추적(89.0점, P0), ②협력사 탄소 데이터 일괄 수집 자동화(81.0점, P1). `content/proposals.md`에 문제정의/근거VoC/개선제안/점수 산출 근거 기록.

- 대안으로 고려했던 것: `output/sample-challenge.html`의 40/40/20 가중치를 그대로 차용하는 방법도 있었으나, 우선순위 기준을 직접 설계해야 하는 채점 항목(양식·우선순위 설계 10점)이므로 근거를 재구성해 비즈니스 임팩트 비중을 더 높이는 쪽으로 차별화.

## [7단계 보강] 가중치 근거 보강 및 민감도 분석 추가
- 결정 내용: PCA·CRITIC·Entropy 같은 데이터 기반 가중치 산출법 도입을 검토했으나 채택하지 않음. 대신 (1) 비즈니스 임팩트·규제 긴급성(선행 지표) vs VoC 빈도(후행 지표)라는 구조적 근거로 가중치 배분 이유를 보강하고, (2) 45/25/30 외 4개 가중치 시나리오(균등, 규제긴급성 최우선, 비즈니스임팩트 최우선, 극단적 단일기준)로 순위 안정성을 검증하는 민감도 분석을 `content/proposals.md`에 추가.
- 기각 이유: PCA/CRITIC/Entropy는 의사결정 행렬의 대안(행) 수가 충분할 때(통상 5~10개 이상) 의미 있는 분산·상관구조를 계산할 수 있는데, 현재 기획안은 2건뿐이라 이 방법들을 적용하면 통계적으로 무의미한 가중치가 나온다. 또한 각 기준의 원점수(비즈니스 임팩트 85점 등) 자체가 이미 정성적 판단값이므로, 가중치만 객관화해도 "주관적 원점수 × 정교해 보이는 가중치"라는 겉보기 정합성만 얻을 뿐 실질적 객관성은 확보되지 않는다고 판단.
- 민감도 분석 결과: 비즈니스 임팩트를 70%까지 높여도 CBAM 기획안의 P0 순위는 유지되며, 순위가 뒤집히는 것은 비즈니스 임팩트 100% 단일 기준(비현실적 가중치)일 때뿐임을 확인. 원안(45/25/30)의 결론이 특정 가중치 선택에 취약하지 않다는 근거로 리포트에 반영.
- 대안으로 고려했던 것: PCA/CRITIC/Entropy를 실제로 적용해보고 결과를 참고용으로만 병기하는 방법도 검토했으나, n=2에서 나온 결과를 병기하는 것 자체가 "객관적 방법론을 오용했다"는 오해를 살 수 있어 완전히 배제하고 시나리오 기반 민감도 분석으로 대체.

## [8단계] 최종 산출물 생성 및 Skill 패키징

## [8단계 보강] 자동 분류 vs 수동 분류 사용 원칙을 리포트에 명시
- 결정 내용: `content/weekly-report-design.md`(→ `voc-report.md`의 "[Standard] 파이프라인 사용법" 섹션)에 "왜 이 리포트의 분류표는 자동 분류 결과가 아닌가"를 설명하는 문단을 추가. "정답 라벨(type)이 있으면 신뢰하고, 없으면 예측한다"는 원칙과, 자동 분류(97.7%, id 038 불일치)가 실제로 쓰이는 지점(정답 없는 원본/더미 검증, 향후 신규 VoC)을 명시.
- 선택한 이유: 리포트 곳곳에 "파이프라인 재현성 97.7%"를 강조하면서 정작 채점용 분류표는 수동 값을 쓰는 것이, 사전 설명 없이는 "자동화한다면서 결과는 수동"이라는 오해를 살 수 있다고 판단. 자동/수동 분류가 "더 나은 쪽을 고르는" 관계가 아니라 정답 유무에 따라 역할이 분리된 것임을 리포트 내에서 먼저 밝혀 투명성을 높임.
- 대안으로 고려했던 것: 별도 언급 없이 정제 로그 한 줄("검증된 type 컬럼이 있어 규칙 기반 분류 대신 해당 값을 사용")로만 남기는 방법도 있었으나, 이 로그는 사람이 md 파일을 끝까지 읽어야 발견되는 위치라 눈에 띄지 않는다고 판단해 [Standard] 섹션 서두에 명시적 문단을 추가.

## [9단계] 제출 기준 전환 및 Streamlit 허용
- 결정 내용: 현장 안내로 Streamlit 배포가 허용됨에 따라, 최종 제출 완성도 판단 기준을 CLAUDE.md의 100/130/160점 채점표에서 **problem.md의 제출 가이드(기본/권장/도전)** 로 전환. Streamlit 대시보드를 권장 항목 충족 수단으로 추가 구축하기로 결정(10단계). 구글시트 자동 기입 연동(도전 항목)은 이 환경에 Cowork 커넥터·구글시트 MCP 도구가 모두 없어 범위에서 제외.
- 선택한 이유: 9단계 자가 점검에서 발견한 "노션/구글독스 공개 링크(10점)" 갭은 CLAUDE.md 채점표에만 있는 항목이라, 기준을 problem.md로 전환하면서 자연히 해소됨(problem.md 기본 항목은 output/ 폴더 저장만 요구).
- 대안으로 고려했던 것: 노션/구글 독스에 직접 게시해 CLAUDE.md 기준까지 충족하는 방법도 있었으나, 사용자가 명시적으로 "지금 신경 쓸 사항이 아니"라고 판단해 기각. 구글시트 연동은 사용자가 서비스 계정 키를 직접 발급하는 방법도 검토했으나 범위 제외를 선택.

## [10단계] Streamlit 대시보드 구축 (로컬 실행)
- 결정 내용: `dashboard/app.py` 단일 파일로 구성. `output/voc-classification.csv`(검증된 44건 분류)를 데이터 소스로 삼고, `content/insights.md`·`content/proposals.md`·`content/decisions-summary.md`·`content/weekly-report-design.md`를 그대로 `st.markdown()`으로 렌더링해 voc-report.md와 동일한 내용을 대시보드 형태로 재구성. 별도 파싱 로직 없이 기존 마크다운 콘텐츠를 재사용해 이중 관리(리포트/대시보드 내용 불일치) 위험을 없앰.
- 검증: `streamlit run dashboard/app.py --server.headless true`로 로컬 구동 확인(HTTP 200, 로그에 에러 없음), pandas 데이터 처리 로직(유형 집계·필터·검색)을 별도로 실행해 9/12/8/15건 집계가 정확히 재현됨을 확인. 테스트 후 프로세스 정상 종료.
- 선택한 이유: problem.md "권장" 항목(Streamlit 대시보드 또는 HTML)을 HTML에 더해 추가 충족. 리포트(md/html)와 별도 콘텐츠를 만들지 않고 동일 마크다운 소스를 재사용해 8단계에서 이미 확정한 "검증된 분류 사용" 원칙과 자동/수동 분류 투명성 문구를 대시보드에도 그대로 반영.
- 대안으로 고려했던 것: 대시보드 전용으로 인사이트·기획안 텍스트를 다시 작성하는 방법도 있었으나, 소스가 두 곳으로 나뉘면 한쪽만 수정했을 때 불일치가 생길 위험이 있어 기각. 배포(GitHub+Streamlit Cloud)는 사용자가 로컬 실행까지만 범위로 확정해 이번 단계에서 제외.

## [10단계 보강] 대시보드 레이아웃을 리포트 텍스트 재사용에서 카드형 구조로 재설계
- 결정 내용: 최초 버전은 `content/insights.md`·`content/proposals.md`를 `st.markdown()`으로 그대로 흘려보내 사실상 "리포트를 옮겨놓은 것"에 불과했다는 지적을 받아, 인사이트(3절)와 Challenge 기획안(5절)만 `st.container(border=True)` 카드 구조로 재설계. 기획안 카드는 `output/sample-challenge.html`의 구성(제목/우선순위 배지·점수, 문제정의·개선제안·근거VoC·우선순위 점수 4분할)을 Streamlit 네이티브 컴포넌트(columns, progress, 배지용 최소 HTML)로 재현. 인사이트 카드는 제목+근거 건수 배지+근거 id 배지+비즈니스 의미 요약을 먼저 보여주고 전체 근거 문단은 `st.expander`로 접어둠.
- 우선순위 슬라이더로 가중치를 실시간 재계산하는 기능과 유형별 통계 카드-표 클릭 연동 기능은 사용자 지시로 범위에서 제외. 대신 실무자가 회의 중 3초 안에 핵심(우선순위 점수, 근거 건수)을 스캔하고 필요할 때만 펼쳐 전체 근거를 확인할 수 있게 하는 데 집중.
- 근거 VoC id를 기획안·인사이트 카드에 명시적으로 노출한 이유: 실무자가 회의에서 "왜 이게 P0인지" 반박에 대응하거나 근거를 검증할 때, 2절 분류표까지 거슬러 올라가지 않고 바로 어떤 id들이 근거인지 확인할 수 있게 하기 위함(단, id 클릭 시 표 필터링 연동은 이번 범위에서 제외).
- 검증: 헤드리스 실행(HTTP 200, 로그 에러 없음)으로 재확인, 테스트 프로세스 정상 종료.
- 대안으로 고려했던 것: 우선순위 가중치 슬라이더(실시간 재계산)와 통계 카드↔분류표 클릭 필터 연동도 검토했으나, 사용자가 카드형 레이아웃(1번)과 인사이트 카드(4번) 방향만 채택하기로 결정해 이번 범위에서 제외.

## [10단계 보강 2] CS팀 즉시 대응 목록 + 제품팀 반복 기능요청 TOP3 추가, 의사결정 로그 섹션 제거
- 결정 내용: `content/weekly-report-design.md`에서 이미 설계했던 CS팀 관점("즉시 대응 목록")과 제품팀 관점("반복 기능요청 TOP3")을 실제 대시보드 기능으로 구현. (1) 불만 유형 중 고위험 키워드(긴급/마비/반려/누락/위험) 포함 건을 "긴급" 배지로 상단 정렬해 표시. (2) 기능 요청 유형의 `keyword_hint`를 콤마 기준 분리·집계해 빈도 TOP3와 근거 id를 표시. 두 섹션을 분류표(2절) 바로 아래 2단 컬럼으로 배치. "4. 의사결정 로그 요약" 섹션은 실무자 활용도가 낮다고 판단해 대시보드에서 제거(HTML 리포트에는 유지). 인사이트 카드(3절→5절)는 세로 나열에서 `st.columns()` 가로 배치로 변경.
- 선택한 이유: 사용자 질의("실무자들이 바로 인사이트를 얻어갈 수 있는가")에 냉정하게 답한 결과, 기존 대시보드는 이미 결론 낸 인사이트·기획안을 보여줄 뿐 실무자가 스스로 "오늘 뭘 먼저 처리해야 하는지(CS)", "내가 검증해도 결과가 같은지(제품)"를 확인할 수단이 없었음. weekly-report-design.md에 이미 설계돼 있었지만 구현되지 않은 두 관점을 대시보드 기능으로 실제 구현해 격차를 해소.
- 대안으로 고려했던 것: 주간 KPI 증감률(전주 대비)도 weekly-report-design.md에 설계돼 있었으나, 현재 데이터가 단일 시점 스냅샷(44건)이라 "전주 대비"를 계산할 기준 데이터가 없어 구현 대상에서 제외 — 다음 주 데이터가 누적되면 추가 가능.
- 검증: 위험 키워드 로직을 별도 pandas 스크립트로 검증(id 003·010·026이 긴급으로 정확히 플래그됨), keyword_hint 빈도 집계도 별도 검증(Scope3 3건 1위). 헤드리스 실행(HTTP 200, 로그 에러 없음)으로 대시보드 전체 재확인.

## [10단계 보강 3] 대시보드 디자인/레이아웃 리디자인 + 검증 방법 보완
- 결정 내용: `.streamlit/config.toml` 테마(배경 #f4f5f7, 텍스트 #1a1a2e, 포인트 #4361ee)를 추가하고, `dashboard/app.py`에 전역 CSS(`CUSTOM_CSS`)를 주입해 카드(그림자·둥근 모서리), 섹션 제목(색상 좌측 바), 배지, 점수 바를 `output/sample-challenge.html`과 톤을 통일. `st.container(border=True)`+개별 위젯 조합 대신, 각 카드(인사이트 5절, 기획안 6절, CS/제품 3·4절)를 완결된 HTML 블록 하나로 구성해 `st.markdown(unsafe_allow_html=True)`로 렌더링하는 방식으로 전환(Streamlit 내부 미문서화 테스트id에 의존하지 않기 위함 — 사전에 `stVerticalBlockBorderWrapper` 같은 클래스명이 실제로 존재하는지 설치된 패키지의 정적 번들을 grep해 확인했으나 발견하지 못해 신뢰할 수 없다고 판단, 대신 직접 만든 HTML 카드로 완전한 스타일 제어권 확보). 유형별 통계 카드(1절)도 `st.metric` 대신 유형별 포인트 컬러가 적용된 커스텀 스탯 카드로 교체. 인사이트 카드의 "근거 상세 보기"는 `st.expander` 대신 순수 HTML `<details><summary>`로 대체해 카드 안에 자연스럽게 포함되도록 함.
- 검증 방법 결함 발견 및 수정: 기존까지 "헤드리스 실행 + curl로 HTTP 200 확인"을 스크립트 정상성 검증으로 써왔는데, 의도적으로 `raise RuntimeError`를 추가해 재현한 결과 **브라우저(웹소켓) 연결 없이는 Streamlit이 스크립트를 실행하지 않아, HTTP 200과 로그에 에러가 전혀 나타나지 않음**을 확인 — 즉 지금까지의 curl 기반 검증은 "서버 프로세스가 떴다"만 증명했을 뿐 "스크립트가 에러 없이 끝까지 실행된다"는 보장은 아니었음. 이후 Streamlit 공식 테스트 API(`streamlit.testing.v1.AppTest`)로 전환해 `at.run()` 후 `at.exception`이 비어있는지로 실제 스크립트 실행 오류를 검증하는 방식으로 교체. `use_container_width` deprecation 경고(폐기 기한 2025-12-31 이미 경과)도 이 과정에서 발견해 `width="stretch"`로 수정.
- 선택한 이유: 디자인 변경은 특히 f-string 내 HTML 조합처럼 런타임에만 드러나는 문법 오류 위험이 크므로, 이번 기회에 검증 방법 자체의 허점을 바로잡는 것이 이후 반복 작업의 신뢰도를 높인다고 판단.
- 대안으로 고려했던 것: Streamlit 내부 CSS 클래스(`data-testid`)를 리버스 엔지니어링해 `st.container(border=True)`를 그대로 스타일링하는 방법도 시도했으나, 설치된 1.58.0 정적 번들에서 해당 문자열을 찾지 못해(버전 간 계속 바뀔 수 있는 미문서화 API) 안정성이 낮다고 판단해 기각.
